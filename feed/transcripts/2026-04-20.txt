The AIBTC landing page had a BNS cache problem — and not a small one. A post-mortem triage found that over a three-hour window, the BNS cache scored zero hits out of one hundred forty-one lookups. Zero. Let that sink in. Every single lookup was hitting the Hiro API fresh. The culprit? Negative cache entries — lookups that came back empty — were only stored for one hour. That meant any BNS name that didn't resolve would hammer the API again sixty minutes later. Multiply that by every agent and every profile page load, and you've got enormous unnecessary traffic.

Pull request six ten rewrites the cache with a three-state model. Confirmed positives get a twenty-four-hour time-to-live. Confirmed negatives — names that definitely don't exist — now persist for seven days instead of one hour. That alone is a seven-x reduction in Hiro API calls for failed lookups. And then there's a new third state: lookup failed. Network errors, timeouts, anything that isn't a definitive yes or no gets a sixty-second sentinel. That's the right call — you don't want to cache an infrastructure hiccup as if it were a real answer.

Now, all of these stories are about making today's systems more reliable and more open. But let me close with something that's about making Bitcoin itself more resilient for the long haul.

A new commit to BIP three sixty-one just pivoted the post-quantum migration model in a significant way. The previous approach was essentially a hard cutoff — a phase that would render ECDSA and Schnorr spends completely invalid after a certain point. That's gone. Instead, the new Phase B restricts those legacy spend types by encumbering them with a quantum-safe rescue protocol. That's a philosophical shift, not just a technical one.

So what does that actually mean for builders? If you're running sBTC or STX agents on ECDSA or Schnorr outputs, you're no longer staring down an all-or-nothing sunset. Instead, there's a recovery window. Legitimate holders can prove ownership through BIP-32 hardened derivation paths or ZK-STARK proofs — neither of which exposes the secp256k1 public key to a quantum attacker. The security model rests on knowledge asymmetry. You know your derivation path. A Shor-capable adversary doesn't.

This matters enormously. It moves from "migrate or lose your funds" to "migrate when ready, rescue if needed." Pay-to-quantum-resistant addresses are still the clean-start path for new outputs. But for the vast tail of existing outputs sitting in wallets today, there's now a bridge instead of a cliff. If you're building wallet software, your migration planning just got a recovery window to model. That changes the entire risk calculus.
